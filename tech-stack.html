Technical Architecture Blueprint

This production-ready blueprint defines the backend stack, AWS deployment model, security controls, audit strategy, capacity assumptions, and delivery guardrails for launching a high-capacity digital lending platform in Mongolia.

1. Executive Strategy

The launch target is not a hobby deployment or a low-risk pilot; it is a regulated fintech platform expected to survive real payment-cycle surges.

Current Stage

The platform uses a managed hybrid cloud topology on AWS from launch. Application compute remains simple and containerized, while durable state, secrets, audit evidence, and database activity tracking are isolated into services designed for stronger durability, operational visibility, and controlled access.

The key design principle is direct: financial records must outlive host failures, deployment mistakes, queue stalls, and provider downtime. Redis, RabbitMQ, application logs, and third-party provider traces are supporting systems. PostgreSQL remains the source of truth, immutable audit storage preserves compliance evidence, and asynchronous integration paths prevent slow external systems from consuming the API tier.

Decision Area Current Blueprint Position Critical Reason
Data tier Managed hybrid cloud topology on AWS from launch, with primary PostgreSQL on AWS RDS. Reduces data-loss exposure, disk I/O contention, backup burden, and failover risk for the lending ledger.
Secrets and configuration AWS Secrets Manager injection into application memory at startup. Supports controlled access, secret rotation, IAM boundaries, and removal of static secrets from server files.
Operational logging Structured JSON logs streamed to Grafana Loki or AWS CloudWatch. Application hosts must be disposable. Local logs cannot be the only operational evidence.
Immutable audit Amazon S3 bucket with Object Lock enabled for WORM-compliant financial/security events. Compliance evidence must be tamper-resistant and independent from app-server lifecycle.
Deployment topology Phase 0 EC2 app node plus managed RDS; Phase 1 adds gateway, two stateless app nodes, ElastiCache, and Amazon MQ. Separates delivery simplicity from state durability. Compute can stay simple while the ledger is managed.

Managed state boundary

The database, audit trail, secrets, and core operational evidence are treated as protected state. Application hosts can be replaced without becoming the only place where critical records exist.

What is still intentionally simple

The application remains a modular monolith. The blueprint avoids premature microservices because distributed ownership and cross-service transactions would slow v1 delivery and increase failure modes.

Decision stance

Keep code deployment simple, but make durable state, secrets, and audit evidence cloud-managed from the beginning.

2. Market Load and Capacity Targets

The deployment target is Mongolia, where digital finance adoption is already high. Capacity planning should assume salary-cycle bursts and national-holiday traffic rather than smooth average usage.

Capacity Model

Verified market signal

Official LendMN wording currently states "over one million registered users" and coverage of more than 80% of Mongolia's economically active population. Use this as verified context, not an exact capacity number.

Planning assumption

The 1.3M to 1.5M registered-account target and 1.3M to 1.38M competitor benchmark are aggressive planning assumptions unless a primary source is attached. The system should still be load-tested against them.

The platform must be sized for a nationwide consumer-finance usage pattern. Standard off-peak usage is not the limiting case. The dangerous windows are daily commute/lunch peaks, semi-monthly salary dates, and major holiday demand spikes around Tsagaan Sar and Naadam. During those windows, thousands of users may attempt loan drawdowns, repayment, QR merchant payments, and settlement checks at the same time.

The architecture therefore targets flash-spike survival before perfect steady-state efficiency. That is why the blueprint prioritizes database IOPS, bounded connection pools, asynchronous provider calls, row-level locking, idempotency, dead-letter handling, and immutable audit trails.

A platform can pass average-load tests and still fail on payday. The acceptance test must include burst traffic, lock contention, and provider slowness at the same time.

Target behavior

Users should receive immediate, bounded responses while long-running provider synchronization moves through recoverable background processing.

3. Bounded Context Isolation and System Topography

The backend uses a Spring Boot modular monolith: one deployable JVM, one relational source of truth, strict package boundaries, and asynchronous integration boundaries for third-party side effects.

Modular Monolith

The modular monolith choice optimizes for fast delivery without introducing distributed-data failure modes too early. Loan origination, repayment, KYC, merchant payments, and provider synchronization all need transactional clarity. Splitting these into independent services before the domain is stable would create cross-service consistency questions, distributed tracing requirements, and multi-team operational overhead.

The monolith is only acceptable if boundaries are enforced. Each bounded context must expose clear interfaces, avoid direct imports into unrelated modules, and keep financial write paths explicit. The outbox pattern is the required escape valve for operations that cannot safely complete inside the user-facing HTTP transaction.

Auth and Audit Context

Handles stateless token processing with Spring Security 7.x and JJWT 0.12.6. Inbound API requests decode token signature claims in memory to avoid database pool exhaustion during login rushes.

KYC and Customer Context

Controls identity matching through state registries such as DAN or GBD and biometric scanning. Network-bound identity work must run behind asynchronous collection boundaries.

Loan and Repayment Context

Owns the financial ledger core: credit draws, interest tracking, settlements, repayments, and loan state transitions. High-contention operations use explicit row locking.

QPay and Polaris Boundary

Connects to banking and payment networks. User response time must not depend on third-party stability, so provider side effects run through transactional outbox and queue workers.

The backend stack is version-locked around Java 21 LTS, Spring Boot 4.0.6, Spring Framework 7.0.x, Gradle 9.x, PostgreSQL 17, and the operational libraries needed for security, persistence, resilience, and observability.

Virtual threads

Spring Boot should enable virtual threads globally so high-concurrency I/O does not require a large platform-thread pool. This helps API scalability only if blocking sections do not pin carrier threads.

Thread pinning rule

Utilities or libraries that hold monitors through blocking I/O can pin virtual threads. Replace high-contention synchronized sections with ReentrantLock or narrower concurrency controls.

spring:
  threads:
    virtual:
      enabled: true

HikariCP sizing

Pool size must be explicit and fixed. Use Little's Law as the starting estimate: Pool Size = Throughput x Average Query Latency. Set minimum-idle equal to maximum-pool-size to avoid surge-time connection allocation.

Hibernate hardening

Ban implicit lazy loading in high-frequency write paths. Use explicit JOIN FETCH, entity graphs, or batch fetching, and review query counts under realistic request traces.

Provider resilience

Wrap all banking HTTP clients in circuit breakers. After 5 consecutive failures, the breaker should open and return a bounded pending/unavailable response instead of consuming EC2 memory.

Critical runtime position

Virtual threads and fast Graviton CPUs do not compensate for slow queries, oversized transactions, missing indexes, or provider calls inside database transactions. The runtime stack is strong only when the team treats query review, timeout configuration, and pool metrics as release-blocking work.

5. Staged AWS Infrastructure Deployment Playbook

The launch topology uses scalable AWS infrastructure while avoiding Kubernetes-level complexity. Phase 0 separates the primary ledger from the application host. Phase 1 extracts Redis and RabbitMQ into managed HA services and scales the application tier horizontally.

AWS Hybrid

Phase 0: AWS Launch Baseline

0 to 10,000 users. Application compute, transient cache, and worker queues run as isolated Docker containers on one EC2 instance. The primary database is offloaded to managed AWS RDS from launch.

Phase 1: HA Clustered Scale

10,000 to 100,000+ users. Application servers become stateless, Redis and RabbitMQ move to AWS managed services, and public ingress is isolated behind an Nginx gateway.

Phase 0 app-node risk

Phase 0 still has one application EC2 node. That is acceptable only if the business accepts a controlled early-stage compute failure risk and has restore/redeploy automation ready.

Load-test requirement

The claim of 5,000+ RPS must be proven with workload-specific tests, including ledger writes, provider slowness, Redis pressure, RabbitMQ backlog, and database lock contention.

Graviton compatibility

All JVM dependencies, Docker images, native mobile build assumptions, monitoring agents, and provider SDKs must be validated on ARM64/Graviton before production cutover.

6. Security, Compliance, and WORM Audit Trail

The platform handles regulated lending data, customer identity details, repayment activity, and security-sensitive events. Secrets, operational logs, immutable audit records, and database activity tracking must be separated by design.

Compliance

AWS Secrets Manager

Static plaintext passwords or access keys are prohibited in repositories and raw server files. Database passwords, JWT private keys, and external integration tokens are fetched through AWS Secrets Manager and injected into Spring memory at startup.

Separated log paths

Operational logs go to Loki or CloudWatch for debugging and alerting. Critical financial and security events bypass volatile host disks and are written to S3 Object Lock for immutable retention.

Control Blueprint Requirement Why It Matters Critical Caveat
Secrets AWS Secrets Manager API bootstrap for database credentials, JWT keys, and provider tokens. Prevents long-lived secrets from being copied into source code, compose files, shell history, or server env dumps. Must include rotation policy, least-privilege IAM, and startup failure behavior.
Operational logs Structured JSON logs with correlation IDs streamed to Loki or AWS CloudWatch. Supports debugging, alerting, incident review, and system health analysis across disposable app nodes. CloudWatch retention and ingestion cost must be configured deliberately.
Immutable audit Loan approvals, repayments, KYC modifications, and security decisions written to S3 Object Lock. Creates WORM-compliant audit evidence that cannot be altered through normal admin workflows. Object Lock mode and retention mistakes are hard to undo. Configure a non-production rehearsal bucket first.
Database activity pgAudit active on AWS RDS with events piped to CloudWatch. Provides an audit trail independent of application logs, including reads, DDL, and ledger interactions. Audit volume can be large; configure class filters and retention with compliance owners.
PII masking Regex-based scrubbing for phone numbers, registration numbers, PINs, and account numbers before export. Prevents sensitive user data from leaking into monitoring, support, or analytics tools. Masking must be tested against real payload shapes and should not be the only PII control.

    <pattern>%replace(%msg){'(?i)(registration_number|pin|account_no)"\s*:\s*"([^"]+)"', "$1\":\"********\""}%n</pattern>
  </layout>
</encoder>

Audit completeness

An audit trail is only useful if every regulated event is captured. The team must define the event catalog before launch, not infer it after incidents.

Object Lock mode

Governance mode and compliance mode have different administrative properties. The wrong mode or retention period can create operational or legal problems.

IAM blast radius

Application roles should write audit objects but not delete or shorten retention. Human break-glass roles must be logged, rare, and reviewed.

7. Team Delivery, Testing, and Pipeline Governance

The monolith stays healthy only when boundary checks, real-infrastructure tests, queue recovery, mobile-release constraints, and environment parity are enforced by the delivery pipeline.

Governance

Delivery governance is not process decoration. It is how the team prevents local convenience from becoming production risk. ArchUnit protects package boundaries, Flyway protects schema evolution, Testcontainers protects integration confidence, RabbitMQ DLX protects queue progress, and EAS profiles protect mobile environment targeting.

ArchUnit boundary control

Gradle 9.x must run architecture tests that fail the build when modules import unauthorized packages. This is the guardrail that makes a modular monolith real.

RabbitMQ DLX recovery

Every queue path needs a Dead Letter Exchange. If an outbox message fails after 3 automated retries, route it to a DLQ and alert developers without blocking other transactions.

Most dangerous weak assumption

Assuming infrastructure scale alone solves payday spikes. The harder risks are row contention, duplicate requests, third-party latency, and queue recovery.

Most likely team failure

Letting module boundaries blur because a monolith makes direct imports easy. ArchUnit and review standards must reject shortcuts before they become architecture.

Most expensive late fix

Audit and PII mistakes are costly to retrofit. Immutable event taxonomy, masking tests, and retention settings must be designed before real customer data lands.
